# external research: flagged topics

## research topic 1: token bucket algorithm — correct for velocity?

### findings

#### [FACT] token bucket allows burst traffic while enforcing average rate

> "Token bucket is the most practical algorithm for API rate limiting — it allows legitimate burst traffic while enforcing a sustained rate."
— [OneUptime: Token Bucket Implementation](https://oneuptime.com/blog/post/2026-01-30-token-bucket-implementation/view) [practitioner] 2026

> "The token bucket algorithm models rate limiting with a bucket that holds tokens and a refill process that adds tokens at a steady rate."
— [FreeCodeCamp: Token Bucket Rate Limiting with FastAPI](https://www.freecodecamp.org/news/token-bucket-rate-limiting-fastapi) [tutorial] 2025

#### [FACT] leaky bucket enforces strict constant rate — no bursts

> "In the leaky bucket algorithm, packets are added to a queue (bucket), and they are released at a steady, constant rate. If the bucket is full, incoming packets are discarded or queued for later transmission."
— [GeeksforGeeks: Token Bucket vs Leaky Bucket](https://www.geeksforgeeks.org/system-design/token-bucket-vs-leaky-bucket-algorithm-system-design/) [tutorial] 2024

> "Leaky bucket ensures a steady, uniform output rate regardless of the input burstiness."
— [Medium: Token Bucket vs Leaky Bucket](https://medium.com/@apurvaagrawal_95485/token-bucket-vs-leaky-bucket-1c25b388436c) [practitioner] 2024

#### [FACT] slide-window provides fairness over time

> "Sliding window rate limiting solves the boundary problem by considering a continuously moving time window, providing a more accurate and fair distribution of requests."
— [Redis: Sliding Window Rate Limiting](https://oneuptime.com/blog/post/2026-01-25-redis-sliding-window-rate-limiting/view) [official] 2026

> "For most APIs, the sliding window counter offers the best balance of accuracy, simplicity, and low memory usage."
— [Arcjet: Rate Limiting Algorithms](https://blog.arcjet.com/rate-limiting-algorithms-token-bucket-vs-sliding-window-vs-fixed-window/) [practitioner] 2024

#### [FACT] major APIs use token bucket

> "Amazon API gateway throttles requests to the API server using the token bucket algorithm, and Stripe also uses the token bucket algorithm for rate limiting."
— [OneUptime: Token Bucket Implementation](https://oneuptime.com/blog/post/2026-01-30-token-bucket-implementation/view) [practitioner] 2026

#### [OPIN] algorithm choice depends on burst tolerance

> "Choose token bucket when you need flexibility and can handle traffic bursts, and choose leaky bucket when you need predictable, consistent output rates."
— [Medium: Token Bucket vs Leaky Bucket](https://medium.com/@apurvaagrawal_95485/token-bucket-vs-leaky-bucket-1c25b388436c) [practitioner] 2024

> "Want fairness over time? Use Sliding Window. Need to allow bursts? Use Token Bucket. Need predictable pacing? Go with Leaky Bucket."
— [DEV: Rate Limiting Deep Dive](https://dev.to/dylan_dumont_266378d98367/rate-limiting-deep-dive-token-bucket-vs-leaky-bucket-vs-sliding-window-47b7) [practitioner] 2024

### convergence

sources [1], [2], [4], [6], [7] agree: **token bucket is standard for API rate limits** when burst tolerance is desired.

sources [3], [5] agree: **slide-window is better for strict fairness** but heavier.

no conflicts found.

### verdict for with-bottleneck

**[SUMP] token bucket fits velocity** — the wish doc's velocity config (`{ quantity, duration }`) maps directly to token bucket semantics:
- quantity = bucket capacity / refill amount
- duration = refill interval

token bucket allows burst (up to capacity) while it enforces average rate — matches typical API usage patterns (Stripe, AWS).

**alternative considered**: slide-window would be more accurate but heavier. for in-memory v1, token bucket is simpler and sufficient.

## research topic 2: async-sema as implementation base?

### findings

#### [FACT] async-sema is maintained by Vercel, 500k+ weekly downloads

> "The npm package async-sema receives a total of 518,264 weekly downloads and is classified as popular."
— [npm: async-sema](https://www.npmjs.com/package/async-sema) [official] 2024

> "async-sema is a semaphore implementation for use with async and await, following the traditional definition of a semaphore."
— [GitHub: vercel/async-sema](https://github.com/vercel/async-sema) [official] 2024

#### [FACT] async-sema provides both semaphore and rate limiter

> "The package creates a rate limiter function that blocks with a promise whenever the rate limit is hit and resolves the promise once the call rate is within the limit set by rps."
— [GitHub: async-sema readme](https://github.com/vercel/async-sema/blob/main/readme.md) [official] 2024

> "Async-sema manages the semaphore count as a list of tokens instead of a single variable containing the number of available resources."
— [npm: async-sema](https://www.npmjs.com/package/async-sema) [official] 2024

#### [FACT] async-sema has limited config options

> "The timeUnit is an optional argument setting the width of the rate limiting window in milliseconds, with a default of 1000 ms."
— [DeepWiki: RateLimit API](https://deepwiki.com/vercel/async-sema/4.2-ratelimit-api) [practitioner] 2024

> "The uniformDistribution argument enforces a discrete uniform distribution over time."
— [DeepWiki: Rate Limiting](https://deepwiki.com/vercel/async-sema/3.2-rate-limiting) [practitioner] 2024

#### [FACT] p-queue is "feature complete" — no active development

> "The project is feature complete and the maintainers are happy to review pull requests, but don't plan any further development."
— [GitHub: sindresorhus/p-queue](https://github.com/sindresorhus/p-queue) [official] 2024

#### [KHUE] should we wrap async-sema or implement from scratch?

async-sema provides:
- concurrency control via Sema class
- rate limit via RateLimit function

async-sema lacks:
- combined concurrency + velocity in single interface
- HOF wrapper pattern (withBottleneck)
- DI / context extraction

### convergence

sources [1], [2], [3] agree: async-sema is battle-tested for concurrency control.

sources [4], [5] show: rate limit is basic (rps only, not IsoDuration).

### verdict for with-bottleneck

**[SUMP] async-sema could be used as semaphore primitive** — use Sema class for concurrency, build velocity layer on top.

**[OPIN] build from scratch may be simpler** — async-sema's API doesn't align with ehmpathy patterns. custom implementation:
- directly supports genBottleneck config shape
- avoids translation between APIs
- zero dependencies (per wish doc preference)

**recommendation**: evaluate in blueprint phase. if async-sema's Sema class simplifies implementation significantly, use it. otherwise, implement semaphore directly (it's ~50 lines of code).
