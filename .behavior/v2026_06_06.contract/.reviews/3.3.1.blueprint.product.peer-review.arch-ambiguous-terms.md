# blocker.1: Overloaded term 'operation' used for multiple concepts

**rule**: .agent/repo=bhuild/role=behaver/briefs/practices/behavior.execution/architect/rule.forbid.ambiguous-terms.md

**locations**:
- .behavior/v2026_06_06.contract/3.3.1.blueprint.product.yield.md

The term 'operation' is overloaded in the blueprint, referring to both the user functions being controlled by the bottleneck (e.g. 'limit simultaneous operations', 'run function') and the internal architectural components (e.g. 'domain.operations/', 'composes other operations'). Per the rule, each concept needs one canonical term to avoid miscommunication and inconsistent implementation. This should be clarified with distinct terms like 'task' for user functions and 'primitive' or 'factory' for internal ones.

**snippet**:
```markdown

composes other operations (e.g., genBottleneck composes semaphore + throttle)
```
