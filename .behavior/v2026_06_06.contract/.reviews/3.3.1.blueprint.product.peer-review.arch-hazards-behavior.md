# blocker.1: Race condition hazard from read-modify-write on closure state

**rule**: .agent/repo=bhuild/role=behaver/briefs/practices/behavior.execution/architect/rule.forbid.behavior-hazards.md

**locations**:
- .behavior/v2026_06_06.contract/3.3.1.blueprint.product.yield.md

The blueprint describes semaphore and throttle implementations that use closure state with read-modify-write sequences for acquire/release and token management. This directly matches the rule's blocker example of race conditions ('read-modify-write without protection') and creates unpredictable behavior under concurrent access. The design relies on single-threaded JS assumptions (no preemption, await points only) which the rule flags as hazards because they pass tests but fail in production, corrupt state silently, or behave differently under load. Although labeled 'acknowledged hazard', the rule states behavior hazards block merge as they are correctness issues, not style, and must be avoided unless truly unavoidable (with explicit documentation still not removing the blocker status).

**snippet**:
```markdown
### concurrency model (acknowledged hazard)

**hazard**: semaphore and throttle use closure state with read-modify-write sequences.

**why intentional**: this is safe in single-threaded JavaScript:
- no preemption between synchronous operations
- async/await yields at await points only
- acquire() is atomic: check-and-queue happens synchronously before any await

**mitigation**: documented as JS-only; not portable to multi-threaded runtimes.
```

# blocker.2: Non-determinism and time assumption hazard in throttle

**rule**: .agent/repo=bhuild/role=behaver/briefs/practices/behavior.execution/architect/rule.forbid.behavior-hazards.md

**locations**:
- .behavior/v2026_06_06.contract/3.3.1.blueprint.product.yield.md

The blueprint introduces a throttle that depends on wall-clock time for token refill, creating non-determinism and time assumptions explicitly forbidden by the rule. The rule asks: 'does this depend on wall clock time, random values, or external state?' and 'does this use setTimeout/setInterval with time assumptions?' and 'does this poll without backoff or max attempts?'. Production behavior will vary with actual timing, load, and clock skew, leading to intermittent failures that are impossible to debug. Mitigation via jest fake timers only covers tests and does not make the runtime predictable. This is documented but the rule requires predictable behavior under all conditions and classifies such hazards as blockers.

**snippet**:
```markdown
### time dependency in throttle (acknowledged hazard)

**hazard**: throttle refill depends on wall-clock time; non-deterministic in production.

**why unavoidable**: velocity control inherently requires time measurement. this is fundamental to the domain, not a design flaw.

**mitigation**: tests mock time for determinism:
- jest.useFakeTimers() in unit tests
- jest.advanceTimersByTime(ms) to simulate passage
- documented in test scenarios as "(time mocked)"
```

# blocker.3: Order-dependence hazard in schedule acquisition

**rule**: .agent/repo=bhuild/role=behaver/briefs/practices/behavior.execution/architect/rule.forbid.behavior-hazards.md

**locations**:
- .behavior/v2026_06_06.contract/3.3.1.blueprint.product.yield.md

The blueprint specifies that schedule() acquires semaphore before throttle, creating order-dependent behavior. The rule flags order-dependence as a blocker: 'does this assume another function was called first? are there implicit setup/teardown requirements? would a reversed call order break behavior?'. This creates non-obvious control flow where the order is critical for backpressure but hidden from callers. The rationale ('semaphore-first bounds total queue depth') does not eliminate the hazard per the rule; such implicit dependencies cause intermittent failures and are correctness issues that block merge.

**snippet**:
```markdown
### acquisition order (acknowledged hazard)

**hazard**: schedule() acquires semaphore before throttle; order-dependent.

**why intentional**: semaphore-first bounds total queue depth. throttle waiters hold semaphore slots to create backpressure. the alternative (throttle-first) allows unbounded waiters, which defeats the purpose of concurrency limits.

**mitigation**: order is internal to schedule(); callers cannot observe or depend on it. documented here for maintainers.
```
