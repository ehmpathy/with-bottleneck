# blocker.1: maintenance hazard: shared mutable state

**rule**: .agent/repo=bhuild/role=behaver/briefs/practices/behavior.execution/architect/rule.forbid.maintenance-hazards.md

**locations**:
- .behavior/v2026_06_06.contract/3.3.1.blueprint.product.yield.md

The target blueprint plans implementation using closure state with read-modify-write sequences for the semaphore and throttle primitives. This directly introduces shared mutable state, which the rule explicitly forbids as a maintenance hazard because it hides errors, creates race conditions, and makes code hard to reason about. The rule requires use of const (not let), spread/clone (not assignment), and no shared mutable state. The blueprint acknowledges this as a 'hazard' but claims it is 'intentional' due to JS single-threading, which does not comply with the rule. This structural defect will compound over time as debug hours and silent defects. The design must be changed to avoid mutable state entirely (e.g. immutable updates or different concurrency model).

**snippet**:
```md
### concurrency model (acknowledged hazard)

**hazard**: semaphore and throttle use closure state with read-modify-write sequences.

**why intentional**: this is safe in single-threaded JavaScript:
- no preemption between synchronous operations
- async/await yields at await points only
- acquire() is atomic: check-and-queue happens synchronously before any await

**mitigation**: documented as JS-only; not portable to multi-threaded runtimes.
```
