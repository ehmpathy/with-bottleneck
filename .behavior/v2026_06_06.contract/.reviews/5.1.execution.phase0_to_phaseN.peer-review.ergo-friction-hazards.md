# blocker.2: unclear error message for release without prior acquire

**rule**: .agent/repo=bhuild/role=behaver/briefs/practices/behavior.execution/architect/rule.forbid.friction-hazards.md

**locations**:
- src/domain.operations/semaphore/genSemaphore.ts:42
- src/index.acceptance.test.ts:310

When a user (or misbehaving code) calls semaphore.release() without a prior acquire, it throws UnexpectedCodePathError with short message 'release called without prior acquire' and raw details object containing active/hint. This is a user-facing blocked/misuse state exposed via the public BottleneckSemaphore interface. The error does not provide a clean, actionable message in the primary string (what went wrong + how to fix). Per the rule, every error path and blocked state must have clear messages and be acceptance-tested/snapped so reviewers can see the exact user experience.

**snippet**:
```ts
if (state.active === 0) {
      throw new UnexpectedCodePathError(
        'release called without prior acquire',
        { active: 0, hint: 'ensure acquire() is awaited before release()' },
      );
    }
```

# blocker.3: blocked states not clearly communicated to users

**rule**: .agent/repo=bhuild/role=behaver/briefs/practices/behavior.execution/architect/rule.forbid.friction-hazards.md

**locations**:
- src/domain.operations/genBottleneck/scheduleWithLimits.ts:10
- src/domain.operations/genBottleneck.integration.test.ts:55
- src/index.acceptance.test.ts:240

When schedule() blocks (due to concurrency or velocity), the caller simply awaits a pending promise with no indication of the reason or state. The only way to inspect is via exposed semaphore.queued/active or throttle.tokens, but there is no unified way to query 'why am I blocked' or time-until-available. This is ergonomic friction: unclear flows and blocked states. The rule requires blocked states to be clearly communicated and every such path to be acceptance-tested with snapshots so the actual user experience is visible and verified. While some internal test snapshots exist for 'blocked by velocity', the public contract does not surface clear communication.

**snippet**:
```ts
const thirdPromise = bottleneck.schedule(async () => {
  thirdDone = true;
  return 'third';
});

expect(thirdDone).toBe(false);
expect({
  state: 'blocked by velocity',
  thirdDone,
}).toMatchSnapshot();
```
