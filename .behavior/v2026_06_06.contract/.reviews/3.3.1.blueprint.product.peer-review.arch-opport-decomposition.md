# blocker.1: Unclear grain for stateful factories

**rule**: .agent/repo=bhuild/role=behaver/briefs/practices/behavior.execution/architect/rule.prefer.decomposable-architecture.md

**locations**:
- .behavior/v2026_06_06.contract/3.3.1.blueprint.product.yield.md

The blueprint describes genSemaphore and genThrottle as 'stateful factory' (a distinct grain for concurrency primitives with internal state but no external i/o). The decomposable architecture rule requires every operation to have clear grain as transformer (pure compute, no i/o), communicator (raw i/o boundary), or orchestrator (compose leaf operations). Introducing a fourth 'stateful factory' grain creates ambiguity, preventing the intended benefits of decomposition for safe recomposition and evolution. These should be classified consistently as one of the three grains (likely transformers with encapsulated state, since no i/o).

**snippet**:
```markdown
| stateful factory | genSemaphore | unit | creates concurrency primitive |
| stateful factory | genThrottle | unit | creates rate limit primitive |
| composer | genBottleneck | integration | composes: guard + factories |
```

# blocker.2: Schedule composition not extracted as named operation

**rule**: .agent/repo=bhuild/role=behaver/briefs/practices/behavior.execution/architect/rule.prefer.decomposable-architecture.md

**locations**:
- .behavior/v2026_06_06.contract/3.3.1.blueprint.product.yield.md

The codepath tree for the genBottleneck orchestrator shows 'schedule: genSchedule(semaphore, throttle)' as the final composition step, but genSchedule is absent from the filediff tree (no src/domain.operations/genSchedule.ts or equivalent). This indicates the schedule logic (which composes semaphore + throttle with guaranteed cleanup and specific acquisition order) is likely defined inline inside genBottleneck.ts rather than as a separately named, reusable leaf operation. This violates grain clarity for orchestrators (must compose named operations only, no inline logic) and reduces recomposition potential, as the schedule behavior cannot be easily reused or evolved independently.

**snippet**:
```markdown
    └── [step 3: compose and return]
        └── return { semaphore, throttle, schedule: genSchedule(semaphore, throttle) }
```

# nitpick.1: Inconsistent terminology for orchestrators

**locations**:
