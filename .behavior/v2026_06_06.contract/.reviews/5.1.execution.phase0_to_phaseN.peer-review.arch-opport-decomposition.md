# blocker.1: scheduleWithLimits mixes control flow without named leaf operations

**rule**: .agent/repo=bhuild/role=behaver/briefs/practices/behavior.execution/architect/rule.prefer.decomposable-architecture.md

**locations**:
- src/domain.operations/genBottleneck/scheduleWithLimits.ts

scheduleWithLimits acts as an orchestrator composing semaphore acquire, conditional throttle acquire, fn execution and guaranteed release, but implements the flow with inline try/finally and if logic instead of delegating to named leaf operations. This requires mental simulation of the acquire order, conditional, and cleanup to understand the behavior. Per the rule, orchestrators must read as narrative (each line tells what, not how) with no decode-friction inline; mixed grains or unclear composition in orchestrators is a blocker. Extract to named operations such as acquireSlot, acquireTokenIfConfigured, executeAndRelease for recomposition potential.

**snippet**:
```ts
export const scheduleWithLimits = async <T>(input: {
  fn: () => Promise<T>;
  semaphore: BottleneckSemaphore;
  throttle: BottleneckThrottle | null;
}): Promise<T> => {
  // acquire semaphore (blocks if at concurrency limit)
  await input.semaphore.acquire();

  try {
    // acquire throttle token if configured (blocks if at velocity limit)
    if (input.throttle) {
      await input.throttle.acquire();
    }

    // run function
    return await input.fn();
  } finally {
    // always release semaphore slot
    input.semaphore.release();
  }
};
```

# nitpick.1: asConcurrencyLimit and asThrottleOrNull are premature extractions with 1 usage

**rule**: .agent/repo=bhuild/role=behaver/briefs/practices/behavior.execution/architect/rule.prefer.decomposable-architecture.md

- src/domain.operations/genBottleneck/asThrottleOrNull.ts

asConcurrencyLimit (one-line ?? Infinity) and asThrottleOrNull (simple if + genThrottle call) are extracted to named transformers despite single usage in genBottleneck. Per the rule, wait for 3+ usages before extracting (wet over dry); duplication is cheaper than wrong abstraction. These could be inlined in the orchestrator without losing narrative clarity, preserving flexibility for evolution.

**snippet**:
```ts
export const asConcurrencyLimit = (input: {
  concurrency: number | undefined;
}): number => input.concurrency ?? Infinity;
```

# nitpick.2: assertLimitsValid mixes multiple validations in one operation

**locations**:
- src/domain.operations/genBottleneck/assertLimitsValid.ts

assertLimitsValid is a single transformer with ~8 separate if-throw blocks mixing concurrency and velocity validation (positive, integer, duration presence, positive duration). This reduces recomposition potential and single responsibility per operation. Per the rule, operations should have explicit inputs/outputs and single responsibility; decompose into named transformers like assertConcurrencyValid and assertVelocityValid that the main assert can compose.
