## relevant patterns from ehmpathy ecosystem

the wish doc and briefs reference test patterns from ehmpathy packages. these inform (but do not exist in) this repo.

### pattern 1: given/when/then BDD structure

**source**: test-fns package

**cite**: [1] mechanic briefs `rule.require.given-when-then.md`

> "use jest with test-fns for given/when/then tests"
> "use describe() for related logic"
> "use given() for setup scenario"
> "use when() for action"
> "use then() for outcome assertions"

**relevance**: [REUSE] — all with-bottleneck tests will use given/when/then structure from test-fns.

---

### pattern 2: useBeforeAll for shared resources

**source**: test-fns package

**cite**: [1] mechanic briefs `howto.write-bdd.[lesson].md`

> "use useBeforeAll for shared resources instead of let + beforeAll + afterAll"
> "wrap all tests in single describe block"

**relevance**: [REUSE] — bottleneck instances for test scenes will use useBeforeAll.

---

### pattern 3: useThen for shared results

**source**: test-fns package

**cite**: [2] mechanic briefs `rule.require.useThen-useWhen-for-shared-results.md`

> "use useThen and useWhen from test-fns to capture operation results and share across blocks without let"
> "eliminates redundant api calls"
> "avoids let declarations"

**relevance**: [REUSE] — concurrency/velocity tests that check multiple assertions on same operation will use useThen.

---

### pattern 4: integration tests without mocks

**source**: ehmpathy test philosophy

**cite**: [3] mechanic briefs `rule.forbid.integration.mocks.md`

> "integration tests must not use mocks. mocks defeat the purpose of integration tests."
> "mocks lie — they drift from real service behavior"
> "mocks create false confidence"

**relevance**: [REUSE] — with-bottleneck tests will use real bottleneck instances, no mocks.

---

### pattern 5: test coverage by grain

**source**: ehmpathy test philosophy

**cite**: [4] mechanic briefs `rule.require.test-coverage-by-grain.md`

> "transformer: unit test — pure, no deps, fast"
> "communicator: integration test — i/o, real or fake deps"
> "orchestrator: integration test — composition, side effects"
> "contract: acceptance test + snapshots"

**relevance**: [EXTEND] — with-bottleneck has unique grains:
- `genBottleneck` factory: unit tests (config validation)
- `withBottleneck` wrapper: integration tests (async timing behavior)
- semaphore operations: integration tests (concurrent access)

---

### pattern 6: data-driven caselists

**source**: ehmpathy test patterns

**cite**: [5] mechanic briefs `rule.prefer.data-driven.md`

> "prefer data-driven caselist tests when possible"
> "especially unit tests of transformers"

**relevance**: [REUSE] — config validation tests will use TEST_CASES array pattern:
```ts
const TEST_CASES = [
  { description: 'zero concurrency throws', given: { concurrency: 0 }, expect: { throws: true } },
  { description: 'negative concurrency throws', given: { concurrency: -1 }, expect: { throws: true } },
  // ...
];
```

---

### pattern 7: snapshots with assertions

**source**: ehmpathy test philosophy

**cite**: [6] mechanic briefs `rule.require.snapshots.[lesson].md`

> "use snapshots for output artifacts"
> "easier review in prs of produced artifacts"
> "use both snapshot and explicit assertions"

**relevance**: [EXTEND] — may apply to error messages from config validation. explicit assertions primary, snapshots secondary.

---

### pattern 8: failfast on absent resources

**source**: ehmpathy test philosophy

**cite**: [7] mechanic briefs `rule.require.failfast.md` (test section)

> "tests that lack required resources must fail fast, not skip silently"
> "if a test cannot run its intended behavior, it must fail loud"

**relevance**: [REUSE] — timing-sensitive tests should ConstraintError if system clock unreliable.
