# vision: with-bottleneck

## the outcome world

### day-in-the-life

**before**: a developer needs to limit API calls. they `npm install bottleneck` — last published 2019. security alerts pile up. no fixes coming. they wrap their functions manually, juggle semaphore state, forget to release on error paths. rate limits hit anyway because the token bucket math is off.

**after**: `npm install with-bottleneck`. one import, one wrapper. the function signature stays the same. the bottleneck is injected — either at declaration time for simple cases, or from context for per-tenant limits. tests swap in unlimited bottlenecks. production respects rate limits.

### before/after contrast

| before | after |
|--------|-------|
| abandoned dependency | maintained ehmpathy package |
| manual acquire/release | automatic via `withBottleneck` wrapper |
| global state only | DI via context extraction |
| esm-only alternatives | cjs + esm |
| test couples to real limits | test injects mock bottleneck |

### the "aha" moment

the click happens when someone writes:

```ts
const fetchLimited = withBottleneck(fetch, {
  bottleneck: (_input, context) => context.browser.bottleneck,
});
```

and realizes: "oh — different tenants get different rate limits, and my tests don't wait at all."

## user experience

### usecases and goals

| usecase | goal | contract used |
|---------|------|---------------|
| limit external API calls | avoid bans | `withBottleneck` + `genBottleneck` |
| limit DB connection pool | avoid saturation | `genBottleneck({ concurrency: N })` |
| per-tenant rate limits | fair usage | context extraction pattern |
| test without waiting | fast tests | inject unlimited bottleneck |
| manual slot control | complex flows | `semaphore.acquire/release` |

### contract inputs & outputs

**`genBottleneck(config)`**
- input: `{ concurrency?: number, velocity?: { quantity, duration } }`
- output: `Bottleneck` instance with `schedule()` and `semaphore`

**`withBottleneck(fn, options)`**
- input: function + `{ bottleneck: Bottleneck | ((input, context) => Bottleneck) }`
- output: wrapped function with same signature

### realworld usage

```ts
// 1. module-level limit for stripe api
const stripeBottleneck = genBottleneck({
  velocity: { quantity: 100, duration: { seconds: 1 } }
});

const sdkStripe = {
  getCustomer: withBottleneck(
    async (input: { id: string }) => stripe.customers.retrieve(input.id),
    { bottleneck: stripeBottleneck }
  ),
};

// 2. per-tenant limits via DI
interface UsecaseContext {
  bottleneck: Bottleneck;
}

const syncCustomer = withBottleneck(
  async (input: { id: string }, context: { usecase: UsecaseContext }) => {
    // ...implementation
  },
  { bottleneck: (_input, context) => context.usecase.bottleneck }
);

// 3. tests — no waiting
it('syncs customer', async () => {
  const unlimitedBottleneck = genBottleneck({}); // no limits
  await syncCustomer({ id: '123' }, { usecase: { bottleneck: unlimitedBottleneck } });
});
```

### timelines

1. **install**: `npm install with-bottleneck` (seconds)
2. **wrap first function**: ~5 minutes (estimate)
3. **migrate from `bottleneck` npm**: update config keys (`maxConcurrent` → `concurrency`, `minTime` → `velocity`), wrap with `withBottleneck`
4. **add per-tenant limits**: define context type, extract from context in wrapper options

## evaluation

### how well does it solve goals?

| goal | solved? | notes |
|------|---------|-------|
| replace abandoned `bottleneck` | yes | maintained, typed, ehmpathy patterns |
| HOF wrapper pattern | yes | `withBottleneck(fn, options)` |
| DI support | yes | context extraction pattern |
| testability | yes | inject unlimited bottleneck |
| esm + cjs | yes | standard ehmpathy build |

### pros

- zero abandoned dependencies
- fits ehmpathy `(input, context)` pattern
- two clear usecases: static vs context-extracted
- composable with `withRetry`, `withLogTrail`, etc.
- semaphore exposed for advanced use

### cons

- in-memory only (no redis/cluster in v1)
- no automatic retry on rate limit error (compose with `with-retry`)
- learning curve for context extraction pattern (mitigated by docs)

### edgecases and pit of success

| edgecase | pit of success |
|----------|----------------|
| forget to release semaphore | `schedule()` auto-releases even on throw |
| bottleneck not in context | typescript error at compile time |
| concurrent vs velocity confusion | separate config keys, clear docs |
| zero concurrency | validate config, throw helpful error |
| negative velocity | validate config, throw helpful error |

## open questions & assumptions

### assumptions

1. **in-memory is enough for v1** — cluster coordination can come later (wisher-stated)
2. **ehmpathy packages use `(input, context)` pattern** — yes, verified via internal research
3. **`iso-time` provides `IsoDuration`** — yes, verified via npm

### answered [answered]

1. **`genBottleneck({})` behavior** — unlimited. wish doc specifies `@default Infinity` for concurrency, `@default undefined` for velocity. empty config = no limits.
2. **semaphore always present** — required. wish doc interface shows `semaphore:` without `?` — not optional.
3. **iso-time dependency** — yes. wish doc imports `IsoDuration from 'iso-time'` — already decided.

### questions for wisher [wisher]

1. **default machine-wide bottleneck** — export one? wrapper-fns does; unclear if needed here
2. **"velocity" term** — use "rate" instead? velocity is non-standard; rate is familiar
3. **replace wrapper-fns withBottleneck** — or coexist as separate package?

### research [research]

- token bucket algorithm — correct for velocity? or slide-window/leaky bucket?
- async-sema or similar — use as implementation base?

## what is awkward?

### the "bottleneck from context" signature

```ts
bottleneck: (_input, context) => context.usecase.bottleneck
```

the `_input` feels awkward — rarely needed but always present. alternatives considered:

1. **separate option for context-only**: `bottleneckFromContext: (context) => ...`
   - adds complexity, two ways to do same thing
2. **keep as-is**: consistent with other extractors, typescript handles it
   - **chosen**: consistency wins

### naming: `velocity` vs `rate` vs `throughput`

- `rate` is overloaded (rate limit, rate of return, etc.)
- `throughput` implies output measurement
- `velocity` = quantity per time, matches physics intuition
- **chosen**: `velocity` with `{ quantity, duration }` shape

### semaphore as nested object

```ts
bottleneck.semaphore.acquire()
```

vs flat:

```ts
bottleneck.acquire()
```

nested feels verbose but:
- separates concerns: `schedule()` is main API, `semaphore` is escape hatch
- makes advanced usage intentional
- **chosen**: nested, main path is `schedule()` via wrapper

### IsoDuration dependency

adding `iso-time` as dependency vs inline duration handling:

- `iso-time` is small, well-maintained ehmpathy package
- provides standard format across packages
- prevents reinventing duration parsing
- **chosen**: depend on `iso-time`
