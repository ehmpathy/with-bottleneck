# blueprint: with-bottleneck

## summary

build `with-bottleneck` — a velocity and concurrency control package for ehmpathy ecosystem.

**core exports:**
- `Bottleneck` — type for bottleneck instance
- `genBottleneck(limits)` — factory to create bottleneck instances
- `withBottleneck(fn, options)` — HOF wrapper for functions

**algorithm:** throttle for velocity — per 3.1.1.research.external.product.flagged._.yield.md, throttle fits velocity: allows burst (up to quantity) then enforces average velocity, matches AWS/Stripe patterns.

**dependencies:** `iso-time` (for `IsoDuration` type) — per 1.vision.yield.md, verified via npm registry.

---

## algorithm selection rationale

per 3.1.1.research.external.product.flagged._.yield.md:

| algorithm | behavior | fit |
|-----------|----------|-----|
| throttle | allows burst, enforces average velocity | **chosen** |
| leaky bucket | strict constant velocity, no bursts | rejected — too restrictive |
| slide window | fairness over time, heavier computation | rejected — overkill for v1 |

**decision**: throttle. the wish doc's `{ quantity, duration }` maps directly to throttle semantics (quantity = tokens, duration = refill interval). major APIs (Stripe, AWS) use throttle.

---

## implementation approach rationale

per 3.1.1.research.external.product.flagged._.yield.md:

> "should we wrap async-sema or implement from scratch?"

**decision**: implement from scratch.

| option | pros | cons |
|--------|------|------|
| wrap async-sema | battle-tested | API mismatch, rps-only velocity limit |
| from scratch | fits ehmpathy patterns, zero deps | must implement ~50 lines |

**rationale**: async-sema's API doesn't align with ehmpathy patterns. custom implementation directly supports genBottleneck limits shape, avoids translation, zero dependencies.

## decomposition rationale

### why separate semaphore and throttle?

per 3.1.3.research.internal.product.code.prod._.yield.md pattern 3 [REUSE]: factory function pattern with closure for state management (from simple-in-memory-queue).

| concern | component | responsibility |
|---------|-----------|----------------|
| concurrency | genSemaphore | limit simultaneous operations |
| velocity | genThrottle | limit operations per time |

this separation allows:
- use semaphore-only when no velocity needed
- test each concern independently
- clearer mental model

### why schedule wraps both?

`bottleneck.schedule` composes semaphore + throttle with guaranteed cleanup.

**composition order:**
1. acquire semaphore slot (blocks if at concurrency limit)
2. acquire throttle token if throttle configured (blocks if at velocity limit)
3. run function
4. release semaphore slot (always, even on error)

rationale for order: see "acquisition order (acknowledged hazard)" below.

### edge case: release without acquire

semaphore.release() called without prior acquire throws UnexpectedCodePathError. this fails fast on misuse rather than silently accept invalid state. error includes details: `{ active: 0, hint: 'release called without prior acquire' }`.

### concurrency model (acknowledged hazard)

**hazard**: semaphore and throttle use closure state with read-modify-write sequences.

**why intentional**: this is safe in single-threaded JavaScript:
- no preemption between synchronous operations
- async/await yields at await points only
- acquire() is atomic: check-and-queue happens synchronously before any await

**mitigation**: documented as JS-only; not portable to multi-threaded runtimes.

### time dependency in throttle (acknowledged hazard)

**hazard**: throttle refill depends on wall-clock time; non-deterministic in production.

**why unavoidable**: velocity control inherently requires time measurement. this is fundamental to the domain, not a design flaw.

**mitigation**: tests mock time for determinism:
- jest.useFakeTimers() in unit tests
- jest.advanceTimersByTime(ms) to simulate passage
- documented in test scenarios as "(time mocked)"

### acquisition order (acknowledged hazard)

**hazard**: schedule() acquires semaphore before throttle; order-dependent.

**why intentional**: semaphore-first bounds total queue depth. throttle waiters hold semaphore slots to create backpressure. the alternative (throttle-first) allows unbounded waiters, which defeats the purpose of concurrency limits.

**mitigation**: order is internal to schedule(); callers cannot observe or depend on it. documented here for maintainers.

## test coverage

per 3.1.3.research.internal.product.code.test._.yield.md:
- given/when/then BDD structure from test-fns
- integration tests without mocks
- test coverage by layer (adapted for bottleneck components)

### glossary

| term | definition |
|------|------------|
| **acquire** | claim a resource (slot or token); blocks until available |
| **active** | count of semaphore slots currently held |
| **backpressure** | mechanism to limit queue growth; blocks new requests when at capacity to prevent unbounded memory growth |
| **BDD** | Behavior-Driven Development; test structure with given/when/then from test-fns package |
| **BottleneckSupplier** | type for options.bottleneck: static Bottleneck instance or function that returns one |
| **caselist** | data-driven test input array for parameterized tests; each element defines inputs and expected outputs |
| **concurrency** | max simultaneous operations; enforced by semaphore |
| **composer** | function that composes other operations (e.g., genBottleneck composes semaphore + throttle) |
| **ConstraintError** | error thrown when limits validation fails; includes details for observability (e.g., { field, value, hint }) |
| **context** | second parameter in ehmpathy (input, context) pattern; contains injected dependencies for DI and testability |
| **details** | error metadata object attached to errors (not to be confused with context parameter) |
| **guard** | pure validation function that throws on invalid input; no side effects, no state |
| **HOF** | higher-order function; function that takes or returns a function |
| **IsoDuration** | duration type from iso-time package (e.g., { seconds: 1 }) |
| **scenario** | end-to-end test case in acceptance tests; validates complete use case from start to finish |
| **limits** | the optional parameter to genBottleneck with concurrency and velocity constraints |
| **public boundary** | public entry point in index.ts (exports, sdk boundary) |
| **quantity** | number of tokens available per duration in throttle |
| **queued** | count of operations that await a semaphore slot |
| **refill** | restore tokens to quantity after duration passes |
| **release** | return a semaphore slot; allows next waiter to proceed |
| **schedule** | method that composes semaphore + throttle acquisition around a function |
| **semaphore** | concurrency limiter; controls max simultaneous operations |
| **slot** | one unit of semaphore capacity; held while operation runs |
| **stateful factory** | distinct grain for concurrency primitives: creates closure with internal state but no external i/o; required for semaphores, throttles, queues where state management is the domain responsibility (not a hazard but the fundamental purpose) |
| **throttle** | velocity limiter; controls operations per time via token bucket |
| **token** | one unit of throttle capacity; consumed on acquire, restored on refill |
| **UnexpectedCodePathError** | error thrown when code reaches invalid state that should be impossible; indicates bug not user error |
| **velocity** | rate limit as quantity per duration (e.g., 10 per second) |

|-----------|----------|----------|------|

## type signatures

### Bottleneck

```ts
interface BottleneckSemaphore {
  acquire: () => Promise<void>;
  release: () => void;
  readonly queued: number;
  readonly active: number;
}

interface BottleneckThrottle {
  acquire: () => Promise<void>;
  readonly tokens: number;
}

interface Bottleneck {
  readonly semaphore: BottleneckSemaphore;
  readonly throttle: BottleneckThrottle | null;  // null when velocity not configured
  schedule: <T>(fn: () => Promise<T>) => Promise<T>;
}
```

### genBottleneck

```ts
import { IsoDuration } from 'iso-time';

const genBottleneck: (limits?: {
  concurrency?: number;
  velocity?: {
    quantity: number;
    duration: IsoDuration;
  };
}) => Bottleneck;
```

### withBottleneck

```ts
type BottleneckSupplier<TInput, TContext> =
  | Bottleneck
  | ((input: TInput, context: TContext) => Bottleneck);

const withBottleneck: <TFn extends (input: any, context?: any) => Promise<any>>(
  fn: TFn,
  options: {
    bottleneck: BottleneckSupplier<Parameters<TFn>[0], Parameters<TFn>[1]>;
  },
) => TFn;
```
